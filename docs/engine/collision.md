# Engine: collision

## The rule (from pokered)

The player moves on a **16×16 px grid** (one "cell"). A block (32×32) contains 2×2 cells.
A cell is **walkable iff its representative tile id is in the tileset's passable list**
(`*_Coll`, see [../data-formats/tilesets.md](../data-formats/tilesets.md)).

The representative tile is the **bottom-left 8×8 tile of the cell** (the player's feet).
Within a block's 4×4 tile layout, the four cells' bottom-left tiles are:

```
cell (sx,sy)  ->  block tile index
  (0,0)  TL   ->  4
  (1,0)  TR   ->  6
  (0,1)  BL   -> 12
  (1,1)  BR   -> 14
```

Encoded in `Main.gd` as `SUB = [[4, 6], [12, 14]]`.

## Why bottom-left (provenance)

`engine/overworld/player_state.asm: GetTileAndCoordsInFrontOfPlayer` samples the player's
standing tile at screen coord `(8,9)` and the tile in front at `(8,11)/(8,7)/(6,9)/(10,9)`.
Adjacent cells' coords differ by 2 tiles (16px), at fixed **column-even / row-odd** screen
positions. The player's 16×16 sprite occupies screen tile rows 8–9 / cols 8–9, so `(8,9)`
is the **bottom-left** 8×8 tile of its cell (the feet). `home/overworld.asm:
CheckTilePassable` then tests that tile id against `wTilesetCollisionPtr` (the passable list).

⚠️ **History:** this was first implemented as *top-left* (`[[0,2],[8,10]]`), which looked
right in Pallet Town at a glance but mis-handled blocks whose upper/lower halves differ: it
**blocked doors and some grass** ("blocked where I should pass") and let you **walk onto the
top row of water**. Fixed to bottom-left. Compare `build/preview/coll_topleft.png` vs
`coll_bottomleft.png` (regenerated by the collision-diff script). Regression checks live in
`Main.gd: _selftest` (door/grass walkable, water blocked).

## Collision grid (`Main.gd`)

`collision` is a `PackedByteArray` of `gw*gh` (cells), `gw=width*2, gh=height*2`,
`1 = walkable`. `is_walkable(Vector2i)` bounds-checks then reads it. Out-of-bounds = blocked
(until map connections / borders exist).

## Tile-pair (elevation) collisions (gh #105)

`_tile_pair_blocked(from, to)` mirrors pokered's `CheckForTilePairCollisions`
(`data/tilesets/pair_collision_tile_ids.asm`): a step between two *walkable* cells is still blocked if
the pair of **feet tiles** (from-cell, to-cell — bidirectional) is a listed pair, "to simulate
differences in elevation." CAVERN pairs are floor `$05` ↔ ledge `$20`/`$2A`/`$21` (and water `$14`↔`$05`
while surfing); FOREST has its own set around `$2E`. Hooked into `Player.gd`'s step (a blocked pair bumps)
and the bot's `_pt_plan`. **Boulder pushes get their own tile-pair check, not the player-step one** — pokered's
`CollisionCheckOnLand` checks the sprite in front *before* the player-step tile-pair rule, so a STRENGTH shove
is routed through `try_push_boulder`, which mirrors `CheckForCollisionWhenPushingBoulder`: the push is refused
if the boulder's destination two steps ahead is unwalkable/stairs, occupied, or forms a tile-pair with the
**player's** tile (i.e. an elevation edge between the player and the destination). It is *not* exempt — but
because the check is player-tile↔destination (two apart), it is looser than the adjacent player-step rule.
This fractures caves into ladder-linked pockets; the bot's cave routes are derived on that pocket graph (Mt
Moon, Rock Tunnel, and Victory Road's multi-floor puzzle — see `docs/notes/gh105-victory-road.md`).

**A boulder moves only on the *second* consecutive push** (pokered `TryPushingBoulder`, `BIT_TRIED_PUSH_BOULDER`,
gh #129). The first push toward a boulder in a given direction only arms the flag and bumps in place ("the
player must try pushing twice before the boulder will move"); the same-direction push after it slides the
boulder one tile. After each tile the flag resets (`DoBoulderDustAnimation` → `ResetBoulderPushFlags`), so
**every tile of travel costs two pushes**. Facing away, stepping onto another cell, facing a different boulder,
or a different push direction all reset the count. In the port, `try_push_boulder` returns `true` only on the
tile-moving push; `_boulder_reset_tried` is called on every turn (`Player.gd`) and every real step
(`_on_player_moved`), and the tried state is keyed to the specific boulder cell + direction so a change of
either re-arms. The bot's `_pt_push_boulder` therefore issues two `_pt_step`s per tile (the derived boulder
*routes* — which cells each boulder ends on — are unchanged; only the input count per tile doubles).
